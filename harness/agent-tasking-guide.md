# 에이전트 사용 가이드

이 하네스는 사용자가 복잡한 에이전트 이름을 몰라도 쓸 수 있게 구성되어 있습니다. 아래 명령 중 하나만 Codex에게 말하면 됩니다.

자세한 내부 라우팅 규칙은 `harness/simple-commands.md`에 있습니다.

## 가장 자주 쓰는 명령

### 1. 오늘 분석해줘

```text
오늘 분석해줘
```

하는 일:

- Alpaca watchlist를 기준으로 종목을 봅니다.
- Alpaca MCP 데이터와 현재 웹 리서치를 함께 사용합니다.
- 종목별 일별/주별/월별 트렌드를 봅니다.
- 추천 후보와 리스크를 정리합니다.
- 결과를 `wiki/`에 저장합니다.
- 주문은 제출하지 않습니다.

### 2. 관심종목 분석해줘

```text
관심종목 분석해줘
```

하는 일:

- Alpaca watchlist에 있는 종목만 리서치합니다.
- 각 종목의 thesis, 촉매, 리스크, 트렌드를 업데이트합니다.
- 원천 자료를 `wiki/evidence-store/sources/`에 남깁니다.
- 주문은 제출하지 않습니다.

### 3. 특정 종목 분석해줘

```text
AAPL 분석해줘
```

여러 종목도 가능합니다:

```text
AAPL MSFT NVDA 분석해줘
```

하는 일:

- 지정한 티커만 리서치합니다.
- 종목끼리 비교하고 순위, 신뢰도, 핵심 리스크를 정리합니다.
- 관련 종목 페이지를 `wiki/research-notes/tickers/`에 업데이트합니다.
- 주문은 제출하지 않습니다.

### 4. 포트폴리오 점검해줘

```text
포트폴리오 점검해줘
```

하는 일:

- Alpaca paper 계좌 상태를 확인합니다.
- 현금, buying power, 포지션, 미체결 주문을 정리합니다.
- `wiki/trade-ledger/positions/current.md`를 업데이트합니다.
- 주문은 제출하지 않습니다.

### 5. 리밸런싱 계획 짜줘

```text
리밸런싱 계획 짜줘
```

하는 일:

- 현재 계좌와 포지션을 기준으로 배분안을 만듭니다.
- `harness/risk-policy.yaml`의 투자, 현금, 종목 한도 규칙을 지킵니다.
- 주문 계획 JSON을 만들고 `scripts/check-risk-policy.py`로 검증합니다.
- 주문은 제출하지 않습니다.

### 6. paper 주문까지 실행해줘

```text
paper 주문까지 실행해줘
```

하는 일:

- 최신 리밸런싱 계획을 확인합니다.
- 필요한 경우 새 리밸런싱 계획을 만듭니다.
- paper mode, 시장 개장, fresh quote, risk gate를 모두 확인합니다.
- 통과할 때만 Alpaca MCP로 paper 주문을 제출합니다.
- 주문 후 바로 거래 후 점검을 실행합니다.

### 7. 거래 후 점검해줘

```text
거래 후 점검해줘
```

하는 일:

- 최근 주문, 체결, 취소, 미체결 주문을 확인합니다.
- 포지션, 현금, buying power를 다시 대조합니다.
- portfolio wiki와 관련 리포트를 업데이트합니다.
- 새 주문은 제출하지 않습니다.

### 8. 거래 회고해줘

```text
거래 회고해줘
```

하는 일:

- 체결된 paper 거래와 현재/청산 포지션을 검토합니다.
- 매수/매도 당시의 종목 분석, 일일 리포트, 주문 계획, raw source를 다시 읽습니다.
- 실제 결과와 비교해 어떤 판단이 좋았고 어떤 판단이 부족했는지 기록합니다.
- 종목별 회고를 `wiki/trade-ledger/reviews/`에 남깁니다.
- 반복적으로 확인된 교훈은 `wiki/policy-book/recommendation-policy.md`에 반영합니다.
- 새 주문은 제출하지 않습니다.

### 9. 위키 정리해줘

```text
위키 정리해줘
```

하는 일:

- 오래된 주장, 누락된 출처, 고아 페이지, index 누락을 찾습니다.
- 모순된 판단을 표시합니다.
- 정리 리포트를 `wiki/research-notes/analyses/`에 저장합니다.
- 주문은 절대 하지 않습니다.

### 10. 과거 기준 추천 시뮬레이션해줘

```text
2026-05-22 기준 추천 시뮬레이션해줘
```

하는 일:

- 과거 특정 날짜 또는 시각 기준으로 당시 알 수 있었던 정보만 사용합니다.
- 당시 위키, raw source, order plan, portfolio 기록에서 후보군을 복원합니다.
- 종목별 `매수`, `보유`, `축소`, `매도`, `관망` 판단과 수량을 제안합니다.
- 결과를 `wiki/backtest-runs/decisions/`에 저장합니다.
- 주문 계획은 dry-run으로만 만들고 실제 주문은 제출하지 않습니다.
- 이후 1D/5D/20D 회고 예정일을 남깁니다.

### 11. 과거 추천 회고해줘

```text
2026-05-22 추천 회고해줘
```

하는 일:

- 과거 추천 시뮬레이션을 실제 이후 가격으로 평가합니다.
- 기본 평가 기간은 1D, 5D, 20D입니다.
- 절대수익률, 벤치마크 대비 초과수익, hit/miss, 수량 적합성을 봅니다.
- 결과를 `wiki/backtest-runs/results/`에 저장합니다.
- 반복적으로 확인된 교훈만 `wiki/policy-book/recommendation-policy.md`에 반영합니다.
- 원래 추천 문서는 수정하지 않습니다.

## 과거 기준 시뮬레이션 회고 사용법

이 기능은 실제 주문을 위한 기능이 아니라, “과거 특정 시점에 내가 어떤 추천을 했어야 했는가”를 실험하고 이후 실제 가격으로 정책을 학습하기 위한 기능입니다. 추천 단계와 회고 단계를 분리해서 미래 정보가 추천에 섞이지 않게 합니다.

### 1. 과거 기준일을 정해 추천 시뮬레이션 만들기

가장 기본 형태:

```text
2026-05-22 기준 추천 시뮬레이션해줘
```

이렇게 말하면 Codex는 2026-05-22 미국 정규장 종가 기준으로 당시 알 수 있었던 정보만 사용합니다. 당시 위키, raw source, 주문 계획, 포트폴리오 기록에 등장했던 종목과 watchlist 후보를 복원하고, 각 종목에 `매수`, `보유`, `축소`, `매도`, `관망` 판단을 붙입니다.

정확한 시각을 지정할 수도 있습니다:

```text
2026-05-22 10:30 ET 기준 추천 시뮬레이션해줘
```

이 경우에는 해당 시각 이전 데이터만 사용해야 합니다. 해당 시각 이후의 가격, 뉴스, 체결 결과는 추천 문서에 들어가면 안 됩니다.

종목군을 좁히고 싶으면 뒤에 조건을 붙입니다:

```text
2026-05-22 기준 추천 시뮬레이션해줘. NVDA AMD AVGO TSM RGTI만 비교해줘.
```

결과는 보통 아래 파일들로 남습니다:

- `wiki/backtest-runs/decisions/YYYY-MM-DD-historical-decision.md`
- `wiki/evidence-store/sources/YYYY-MM-DD-historical-asof.md`
- `wiki/trade-ledger/orders/YYYY-MM-DD-historical-decision.json`

주문 계획 JSON은 수량 추천을 검증하기 위한 dry-run입니다. 실제 paper 주문도 제출하지 않습니다.

### 2. 추천 결과가 평가 가능한 시점에 회고하기

기본 회고 명령:

```text
2026-05-22 추천 회고해줘
```

Codex는 해당 날짜의 `wiki/backtest-runs/decisions/YYYY-MM-DD-historical-decision.md`를 찾아서, 추천 이후 1D/5D/20D 가격 변동을 평가합니다. 회고 단계에서는 미래 가격을 사용해도 되지만, 그 정보는 반드시 별도 회고 문서에만 기록합니다.

결과는 보통 아래 파일로 남습니다:

- `wiki/backtest-runs/results/YYYY-MM-DD-historical-review.md`

회고에서는 다음을 봅니다:

- 추천 라벨이 맞았는지: 매수/보유/축소/매도/관망.
- 절대수익률과 SPY, QQQ, 섹터 ETF 대비 초과수익.
- 추천 수량이 과했는지 또는 너무 작았는지.
- 최대 유리 이동과 최대 불리 이동.
- 당시 정보로 합리적인 판단이었는지.
- 정책에 반영할 만한 반복 교훈이 있는지.

## 뉴스/이벤트 보강 MCP 사용법

주식 분석에서 Alpaca 뉴스만으로 실적, SEC filings, 회사 IR, 애널리스트 맥락, 매크로 지표가 부족하면 `harness/mcp-source-map.md`를 먼저 읽고 보강 MCP를 사용합니다.

등록된 서버는 다음과 같습니다:

- `sec-edgar`: SEC filings, 10-K, 10-Q, 8-K, Form 4, XBRL 재무제표.
- `alpha-vantage`: 실적 캘린더, earnings history/surprise, 기술지표. `ALPHA_VANTAGE_API_KEY` 필요.
- `fred`: CPI, 금리, 실업률, yield curve 같은 매크로 지표. `FRED_API_KEY` 필요.
- `firecrawl`: 회사 IR 페이지, 보도자료, earnings presentation 웹페이지 캡처. `FIRECRAWL_API_KEY` 필요.
- `yahoo-finance`: Yahoo 뉴스, analyst recommendation, holders, insider, stock actions 보조 정보.

사용 순서는 보통 아래와 같습니다:

1. Alpaca MCP로 가격, 뉴스, calendar, corporate action을 확인합니다.
2. `sec-edgar`로 최근 filing과 insider/form 이벤트를 확인합니다.
3. `alpha-vantage`로 실적 캘린더와 earnings surprise를 확인합니다.
4. `fred`로 해당 시점의 매크로 regime을 확인합니다.
5. `firecrawl`로 회사 IR/보도자료 원문을 캡처합니다.
6. `yahoo-finance`로 analyst/news 보조 정보를 확인합니다.

키가 없는 MCP는 억지로 대체하지 말고 raw source의 `데이터 공백`에 남깁니다. 모든 결과에는 조회 MCP, 조회 시각, 티커, 기간, 원천 URL 또는 filing identifier를 기록합니다.

### 3. 정책을 학습시키는 방식

회고 한 번으로 바로 강한 규칙을 만들지 않습니다. 단일 사례는 `wiki/policy-book/recommendation-policy.md`의 `검증 중인 가설`이나 `정책학습 지표`에 남기고, 같은 패턴이 여러 번 반복될 때만 현재 원칙으로 올립니다.

예를 들어 다음 같은 식으로 누적합니다:

- 저신뢰도 고변동 종목을 크게 산 추천이 반복적으로 나빴는지.
- 강한 추세와 충분한 뉴스 근거가 같이 있을 때 매수 추천이 벤치마크를 이겼는지.
- `관망` 처리한 종목의 기회비용이 반복적으로 컸는지.
- 특정 섹터에 과도하게 몰린 수량 추천이 손실 변동성을 키웠는지.

정책 업데이트가 필요한 경우에는 회고 문서 링크, hit rate, 평균 초과수익, evidence count를 함께 남깁니다.

### 4. 추천 명령 예시

```text
2026-05-22 기준 추천 시뮬레이션해줘
```

```text
2026-05-22 기준 추천 시뮬레이션해줘. 당시 보유 종목과 주문 계획 후보만 써줘.
```

```text
2026-05-22 기준 추천 시뮬레이션해줘. 반도체 후보만 보고 SPY QQQ SMH와 비교해줘.
```

```text
2026-05-22 추천 회고해줘
```

```text
2026-05-22 추천 회고해줘. 1D와 5D만 우선 봐줘.
```

### 5. 주의할 점

- 추천 시뮬레이션 단계에서는 미래 가격을 절대 사용하지 않습니다.
- 회고 단계에서도 원래 추천 문서는 수정하지 않습니다.
- 과거 raw source는 사후 수정하지 않고, 새 회고 문서에서만 링크합니다.
- 이 기능은 실제 주문을 제출하지 않습니다.
- 실제 주문이 필요하면 별도로 `paper 주문까지 실행해줘`처럼 명시해야 하며, 그때도 기존 risk gate를 통과해야 합니다.

## 안전한 기본 동작

- 명령에 `주문`, `매수`, `매도`, `실행`이 없으면 주문하지 않습니다.
- 모든 주문은 paper trading만 허용됩니다.
- 미국 주식/ETF만 허용됩니다.
- whole-share day limit order만 허용됩니다.
- 주문 전에는 반드시 `scripts/check-risk-policy.py`를 통과해야 합니다.
- API key나 `.env` 내용은 출력하지 않습니다.

## 추천 사용 흐름

처음에는 이렇게 쓰면 충분합니다:

```text
오늘 분석해줘
```

결과가 괜찮으면:

```text
리밸런싱 계획 짜줘
```

계획까지 괜찮으면:

```text
paper 주문까지 실행해줘
```

주문 뒤에는:

```text
거래 후 점검해줘
```

거래 결과가 쌓이면:

```text
거래 회고해줘
```

주기적으로:

```text
위키 정리해줘
```

정책 학습용 실험을 하고 싶으면:

```text
2026-05-22 기준 추천 시뮬레이션해줘
```

평가 기간이 지난 뒤에는:

```text
2026-05-22 추천 회고해줘
```

## 조금 더 구체적으로 말하고 싶을 때

짧은 명령 뒤에 조건만 덧붙이면 됩니다.

```text
오늘 분석해줘. 반도체 종목을 특히 자세히 봐줘.
```

```text
AAPL MSFT NVDA 분석해줘. 이번 주 뉴스와 실적 관련 리스크를 중심으로 봐줘.
```

```text
리밸런싱 계획 짜줘. 신규 주문 수는 보수적으로 제한해서 제안해줘.
```

```text
포트폴리오 점검해줘. 현재 보유 종목별 비중도 같이 보여줘.
```

사용자는 짧게 말하고, 하네스는 내부적으로 필요한 에이전트와 워크플로우를 선택합니다.
